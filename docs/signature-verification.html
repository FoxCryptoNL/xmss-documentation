The API deliberately does not contain a direct verify_signature function, whereas many other cryptographic signature libraries do. This document explains why this is so; it also provides guidance on how to properly verify a signature in your code.

Problem definition

Different products require different levels of resilience against sophisticated attacks. Although an XMSS signature is very secure, the verification method itself could possibly be bypassed. Security products require a proper security architecture to prevent bypasses of a security function. We identify the following fault injections that an attacker could use to attempt to bypass signature verification:

  • bit errors

    See: Secure boolean functions

  • instruction skipping

    See: Secure conditional branches

Furthermore, some products want to employ Public key obfuscation while others do not.

Solution

The API allows for all scenarios, supporting both the most resilient products as well as those that require a minimum size instead. This is illustrated in the following two examples.

Example: Simple verification

If your product requires neither resilience nor obfuscation, then a possible implementation could be:

// NOTE: no bit error resilience
// NOTE: no instruction skip resilience
// NOTE: no public key obfuscation
bool verify_signature(const XmssPublicKeyBlob *public_key, const XmssBuffer *message, const XmssSignatureBlob *signature)
{
XmssValue256 expected_public_key;
if (xmss_calculate_expected_public_key(&expected_public_key, message, public_key, signature) != XMSS_OK)
{
return false;
}
return memcmp(expected_public_key, public_key->data.public_key, sizeof(XmssValue256)) == 0;
}

Example: Complex verification

If your product requires both resilience and obfuscation, then a possible implementation could be:

void execute_with_verify(const XmssPublicKeyBlob *public_key, const XmssBuffer *message, const XmssSignatureBlob *signature)
{
XmssValue256 expected_public_key;
if (xmss_calculate_expected_public_key(&expected_public_key, message, public_key, signature) != XMSS_OK)
{
// verification error
return;
}
#pragma optimize(off)
if (constant_time_equals(expected_public_key, public_key->data.public_key, sizeof(XmssValue256)) != RESILIENT_TRUE)
{
// verification error
return;
}
execute_random_delay();
if (constant_time_equals(expected_public_key, public_key->data.public_key, sizeof(XmssValue256)) != RESILIENT_TRUE)
{
// verification error
return;
}
execute_random_delay();
if (constant_time_equals(expected_public_key, public_key->data.public_key, sizeof(XmssValue256)) != RESILIENT_TRUE)
{
// verification error
return;
}
#pragma optimize(restore)
// perform operations that require successful verification below this point
}

In this example, the comparison is done with constant time to obfuscate the public key. The example relies on a pre-existing function constant_time_equals. The function constant_time_equals is assumed to return bit error resilient truth values.

In this example, the comparison is done three times with some delay in between, relying on a pre-existing function execute_random_delay. An attacker that attempts to skip instructions must successfully execute at least three well-timed attacks. Note that an attacker may also try to skip the execute_random_delay call itself; further hardening by unfolding the delay operation is possible.

Note that the most time consuming operation, xmss_calculate_expected_public_key(), is called only once, even for this resilient implementation, because an incorrect value in expected_public_key will only lead to verification errors. If, on the other hand, the implementation would have had to call verify_signature multiple times, efficiency would be reduced.

It is beyond the scope of this XMSS library to provide a specific resilient implementation of signature verification; it depends on details of the underlying hardware. It also depends on implementation details of the compiler; optimizations usually need to be turned off and it matters if the branching is on a positive or on a negative verification result. The latter can sometimes be controlled by inverting the condition and using nested if-statements; some compilers support __builtin_expect to control the branching strategy.

In all cases it is necessary to inspect the final assembly code to confirm that the required level of resilience is achieved.

Conclusion

The API allows for both simple (without resilience) and complex (with hardware dependent resilience) signature verification. xmss_calculate_expected_public_key() is resilient by design, without internal complexity. The API also allows for public key obfuscation, while keeping the code size at a minimum for those products that do not require it. This means the XMSS library is compact, efficient, and secure.

For simple verification, extending xmss_calculate_expected_public_key() to verify_signature is trivial.

For complex verification, xmss_calculate_expected_public_key() needs to be called only once for efficiency. The exact details of a resilient comparison implementation is both hardware and compiler dependent, which is outside the scope of this XMSS library.
